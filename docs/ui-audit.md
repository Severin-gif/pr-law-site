# Аудит проекта

## 1. Краткое заключение
Проект находится в состоянии смешанной архитектуры: в одном репозитории одновременно присутствуют Vite SPA (`src/*`), Express-сервер (`server/index.js`) и фрагменты Next App Router (`app/*`, `server/app/*`, `public/sitemap.ts`). Это снижает стабильность релизов (неочевидно, какой runtime является «истиной»), ухудшает повторяемость сборки и блокирует масштабирование без архитектурной очистки.

Ключевые системные проблемы:
- дублирование и конфликт слоёв (routing/layout/api/seo);
- размытые границы ответственности компонентов и инфраструктуры;
- мёртвый код и неиспользуемые ветки реализации;
- отсутствие целостной системы аналитики/SEO-архитектуры для SPA;
- неравномерная UI-система (типографика, карточки, модалки, секции).

Общий уровень качества: **средний/ниже среднего по инженерным критериям** (визуально проект уже оформлен, но архитектурно нестабилен).

## 2. Критические проблемы

### 2.1 Смешение трёх платформ в одном проекте (Vite + Express + Next)
- **Проблема:** одновременно используются несовместимые архитектурные слои.
- **Где:** `src/*`, `server/index.js`, `app/api/*`, `server/app/*`, `public/sitemap.ts`.
- **Почему это плохо:** невозможно однозначно определить production-контур; поддержка и деплой зависят от «ручного знания».
- **Последствия:** регрессии при релизах, сложность масштабирования, высокая вероятность расхождения поведения dev/prod.
- **Что сделать:** выбрать один стек как основной. Для текущего репозитория логичнее оставить `Vite + Express`, удалить/изолировать Next-артефакты (`app/*`, `server/app/*`, `public/sitemap.ts` как TS-модуль для Next).

### 2.2 Нерабочий endpoint для партнёрской формы
- **Проблема:** фронтенд отправляет на `/api/partner-lead`, но сервер такого маршрута не содержит.
- **Где:** `src/components/partners.tsx` (`FORM_ENDPOINT = "/api/partner-lead"`), `server/index.js` (есть только `/api/request-audit` и `/api/lead`).
- **Почему это плохо:** основная CTA-форма страницы «Партнёрам» в проде будет падать.
- **Последствия:** потеря заявок, недоверие пользователей, невозможность трекинга ключевого события.
- **Что сделать:** унифицировать endpoint (либо изменить фронт на `/api/lead`, либо добавить `/api/partner-lead` на сервере с отдельной схемой payload).

### 2.3 Дублирование глобального layout и нарушение семантики main
- **Проблема:** `App.tsx` уже рендерит `<main>`, при этом `Home.tsx` возвращает второй `<main>`.
- **Где:** `src/App.tsx`, `src/pages/Home.tsx`.
- **Почему это плохо:** semantic conflict, ухудшение доступности и предсказуемости структуры.
- **Последствия:** неустойчивость для SEO/assistive tools, рост хаоса при добавлении новых страниц.
- **Что сделать:** оставить единственный `<main>` на уровне layout (`App.tsx`), в страницах использовать `<div>/<section>`.

### 2.4 Broken navigation на внутренних страницах
- **Проблема:** Header содержит якоря `#about/#services/#trust/#contact`, но Header глобальный и отображается на всех маршрутах.
- **Где:** `src/components/layout/Header.tsx`, `src/App.tsx`.
- **Почему это плохо:** на `/about`, `/privacy`, `/terms`, `/partners`, `/pro-bono` якоря не ведут к целевым секциям.
- **Последствия:** падение UX, неочевидные «битые» переходы.
- **Что сделать:** условный рендер навигации по route; либо заменить на `Link to="/#section"` для глобальной консистентности.

### 2.5 SEO-архитектура фактически отсутствует для текущего runtime
- **Проблема:** Next `metadata/sitemap/robots` присутствуют, но приложение работает как Vite SPA + Express.
- **Где:** `server/app/layout.tsx`, `server/app/sitemap.ts`, `server/app/robots.ts`, `public/sitemap.ts`, при этом entrypoint — `src/main.tsx`.
- **Почему это плохо:** метаданные из Next не используются; sitemap в TS-файлах не интегрирован в runtime.
- **Последствия:** нестабильная индексация, недостоверное ожидание SEO-поведения.
- **Что сделать:** реализовать SEO как часть выбранной архитектуры: статические `public/sitemap.xml`, `public/robots.txt`, SSR/пререндер-стратегия или минимум корректные meta/OG на уровне index.html + server headers.

## 3. Замечания средней важности
- Дублирование middleware редиректа `www -> non-www` в `server/index.js` (блок подключён дважды).
- В `ContactSection` форма не имеет обработчика submit/API-интеграции: UI обещает действие, но технически оно не фиксируется.
- Избыточные и неиспользуемые компоненты: `TrustLettersSection.tsx`, `TrustLettersCarousel.tsx`, `RequestReviewModal.tsx`, `AboutSpeakerSection.tsx`, `QuoteSection.tsx`, `TrustLettersTab.tsx`.
- `src/App.css` содержит шаблонный Vite-код и не используется в текущей системе стилей (Tailwind-first).
- Типизация роутера потенциально конфликтна: `react-router-dom@6`, но dev dependency `@types/react-router-dom@5`.
- Модальные окна реализованы вручную в нескольких вариантах без единого базового компонента (разные z-index, обработка Esc, scroll lock).
- Компоненты содержат доменную/сетевую логику прямо внутри UI (`RequestAuditModal`, `partners.tsx`), что снижает переиспользуемость.

## 4. UI/UX-аудит главной страницы

### 4.1 First screen (Hero)
**Что хорошо:**
- сильный визуальный фокус; читаемый H1; очевидный CTA.

**Проблемы:**
- слишком много декоративных слоёв (градиенты/виньетки/тени), что усложняет визуальную систему;
- fixed background через inline-style лишает контроля оптимизации загрузки изображения;
- CTA открывает модалку, но событие аналитики не предусмотрено.

**Инженерное решение:**
- сократить число overlay-слоёв до 1–2;
- перевести hero image в управляемый компонент с `img/picture` и стратегией загрузки;
- добавить обязательные event hooks (`cta_hero_click`, `modal_open`).

### 4.2 Блок услуг
**Текущая реализация:** mobile — горизонтальный snap-slider, desktop — 3-колоночный grid (`ServicesSection`).

**Почему блок выглядит «слабым»:**
- карточка перегружена внутренними уровнями (title + bullets + result-box + link), при этом высота фиксирована `min-h-[360px]` — визуально создаёт «кирпичи»;
- на мобильном ширина карточки `w-[86%]` + плотный текст => ощущение зажатости;
- типографика внутри карточек разнокалиберная (11/15/24/26) без единого вертикального ритма.

**Grid vs Carousel:**
- для 3 услуг приоритетнее **grid + stack на mobile** (1 колонка, затем 2/3), а не carousel;
- carousel оправдан при 6+ карточках или динамическом каталоге.

**Практическое решение:**
- mobile: вертикальный stack (gap + card padding), убрать обязательный horizontal scroll;
- desktop: оставить grid;
- упростить карточку: унифицировать блок «Результат» и CTA, уменьшить глубину визуальных контейнеров.

### 4.3 Другие секции (About / Trust / FAQ / Contact)
- `AboutSection`: хороший narrative, но абсолютное позиционирование кнопки `Подробнее` в карточке повышает риск коллизий текста на узких ширинах.
- `TrustSection`: вкладки уместны, но содержание табов разнотипно (лист, модалки, iframe), что ломает единообразие.
- `FAQSection`: читабельно, но фоновое изображение + затемнения конкурируют с контентом; для «дорогого чистого» вида лучше спокойный однотонный контур.
- `ContactSection`: визуально консистентно, но функционально незавершён (нет отправки, валидации, статуса).

## 5. Layout и компонентная архитектура
- Единая точка каркаса формально есть (`App.tsx`: Header/Main/Footer), но страница `Home` повторно вводит layout-уровень (`<main>`), а вторичные страницы имеют собственные фоновые контейнеры — это частично смешивает page-level и section-level ответственность.
- `Header/Footer` глобальные, но содержат route-специфичное поведение (якоря home), что противоречит идее изолированного layout-компонента.
- В `components` смешаны: layout, страницы (`partners.tsx`), секции, модалки, карусели, доменные блоки. Нужна нормализация по слоям (`components/ui`, `components/sections`, `features/*`).
- Компоненты модалок дублируют базовую инфраструктуру (backdrop, esc, lock-scroll, close semantics) вместо общего примитива.

## 6. Адаптивность

### Что хорошо
- Tailwind breakpoints используются последовательно (`sm/md/lg`).
- Ключевые секции имеют адаптивные отступы и размеры текста.

### Что плохо
- В Header отсутствует мобильная навигация: меню полностью скрывается до `md`.
- Сервисные карточки на mobile реализованы как горизонтальный скролл, что ухудшает сканируемость контента.
- Несколько секций завязаны на «десктопный» visual treatment (крупные фоны + сложные overlay), на mobile это съедает контраст/воздух.
- В `EcosystemCarousel` клавиатурный listener на `window` глобален — риск конфликтов с другими интерактивными зонами.

### Что переделать
- Сделать mobile-first навигацию (бургер/drawer или компактный стек ссылок).
- Для услуг перейти на stack-first; carousel оставить только при реальной перегрузке карточками.
- Свести вариативность декоративных контейнеров к 1–2 паттернам секций.

## 7. SEO и семантика
- Дублирование и рассинхрон sitemap-источников (`public/sitemap.ts`, `server/app/sitemap.ts`) без рабочей интеграции в выбранный runtime.
- Семантический дефект: два `<main>` в дереве главной страницы.
- Отсутствует верифицируемая стратегия metadata/OG/Twitter для SPA-страниц.
- Нет явной schema.org-разметки (Organization/LegalService/FAQPage) для ключевых блоков доверия и FAQ.
- Для Core Web Vitals рисковы тяжелые фоновые изображения в секциях без управляемой оптимизации доставки.

## 8. Технический долг
- Мёртвые компоненты и шаблонные остатки (`App.css`, неиспользуемые section/modal/carousel файлы).
- Разные API-ветки (Express + Next route handlers) без единой точки правды.
- Смешение уровней абстракции: UI-компоненты выполняют submit/validation/integration напрямую.
- Naming debt: `src/components/partners.tsx` фактически является page-компонентом.
- Избыточные зависимости/типы (`@types/react-router-dom@5` при v6).
- Нефиксированные «временные» решения в коде (комментарии «позже подключим API» и т.п.).

## 9. План исправления по приоритетам

### P0 — исправить сразу
1. Выбрать и зафиксировать единый runtime (рекомендуется Vite + Express), удалить/изолировать Next-слой.
2. Починить endpoint партнёрской формы (`/api/partner-lead` vs `/api/lead`).
3. Убрать дублирующий `<main>` в Home.
4. Исправить навигацию Header для внутренних страниц.
5. Ввести минимальный analytics-контур для критичных событий:
   - `form_submit_start`
   - `form_submit_success`
   - `form_submit_fail`
   - `cta_click`
   - `payment_start/success/fail` (заглушка интерфейса событий уже сейчас).

### P1 — ближайший цикл
1. Удалить мёртвый код/неиспользуемые компоненты.
2. Нормализовать структуру компонентов (ui primitives / sections / features).
3. Унифицировать модальные окна через единый базовый компонент.
4. Пересобрать блок услуг в stack-first mobile + grid desktop.
5. Внедрить рабочую SEO-архитектуру для SPA (реальный sitemap.xml, meta policy, schema).

### P2 — после стабилизации
1. Подготовить шаблон клонирования под другие ниши (конфигурируемые секции/контент-модели).
2. Добавить слой интеграций (`analytics`, `payments`, `integrations`) как изолированные модули.
3. Перейти к дизайн-токенам (типографика/spacing/radius/color) через централизованный config.

## 11. Вывод
Текущее состояние мешает достижению требований «стабильность/масштабируемость/повторяемость» не из-за дизайна, а из-за архитектурной неоднозначности и дублирования технических слоёв. Приоритет — не косметика секций, а устранение конфликтов runtime, фиксация layout-контракта, чистка мёртвого кода и внедрение наблюдаемости (analytics + единые события форм).

После выполнения P0/P1 проект можно безопасно масштабировать: подключать ИИ-ассистента как отдельную feature-зону, клонировать под другие ниши через конфигурацию секций и наращивать страницы без переписывания ядра.

## Quick Wins
- Привести `FORM_ENDPOINT` в `partners.tsx` к реально существующему API.
- Удалить второй `<main>` на главной.
- Добавить mobile-меню в Header (сейчас навигация скрыта до `md`).
- Удалить неиспользуемые компоненты и `src/App.css`.
- Убрать дубли редирект-мидлвари в Express.
- Для блока услуг на mobile заменить horizontal slider на вертикальный stack.
- Добавить базовый event logger для CTA и submit-статусов.

## Решения, которые нельзя принимать
- Дублировать layout-слои на уровне страниц.
- Держать в одном прод-контуре одновременно Vite- и Next-архитектуру.
- Оставлять UI-формы без рабочей отправки/трекинга.
- Размазывать page-компоненты по `components/*` без слоистой структуры.
- Добавлять одноразовые модалки/карусели вместо базовых переиспользуемых примитивов.
- Использовать декоративные carousel-паттерны там, где достаточно grid/stack.
- Хранить «временные» решения без явной пометки и срока удаления.
